# Install Guide

[English](../README.md) | [繁體中文](../README.zh-TW.md)

ACG Collector can be installed as a Progressive Web App (PWA) from a supported mobile browser. It still runs as a website, but an installed PWA can open from the home screen and use the app shell after the first successful visit.

## Before You Install
- Use the production app URL: [https://acg-app-steel.vercel.app/](https://acg-app-steel.vercel.app/), not the development server.
- The app should be served over HTTPS in production. Browser install prompts and service workers generally require HTTPS.
- Your collection is stored in browser storage through IndexedDB on the device/browser where you use the app.
- Deleting browser site data, clearing storage, uninstalling the browser profile, or changing devices may remove the local collection.
- Use **Export Backup** before clearing browser data, switching devices, resetting a phone, or trying a new production URL.

## Android Chrome
1. Open the production app URL in Chrome.
2. Wait for the page to finish loading.
3. If Chrome shows an install prompt, choose **Install**.
4. If no prompt appears, open the Chrome menu and choose **Install app** or **Add to Home screen**.
5. Launch **ACG** from the home screen or app drawer.
6. Confirm it opens without the normal browser address bar.

## iOS Safari
1. Open the production app URL in Safari.
2. Tap the **Share** button.
3. Choose **Add to Home Screen**.
4. Confirm the app name and tap **Add**.
5. Launch **ACG** from the home screen.
6. Confirm it opens in standalone web app mode rather than as a normal Safari tab.

## Data and Backups
- Data is local to the browser storage area for the installed app/site.
- Browser storage behavior differs between iOS, Android, Chrome, Safari, and private browsing modes.
- Export a backup JSON file regularly if the collection matters.
- Importing a backup replaces the current local collection after confirmation.

## Troubleshooting

### Install Button Does Not Appear
- Confirm you are using HTTPS in production.
- Refresh the page once and wait for it to finish loading.
- Try the browser menu option: **Install app** or **Add to Home screen**.
- Confirm the production build includes a valid manifest and service worker.
- On iOS, use Safari. Other iOS browsers may not expose the same Add to Home Screen flow.

### Icon Looks Wrong
- Remove the installed app and add it again after the latest deployment finishes.
- Clear the browser cache for the site if the old icon keeps appearing.
- Confirm the deployed site serves `/favicon.svg`, `/apple-touch-icon.png`, `/pwa-192x192.png`, `/pwa-512x512.png`, and `/pwa-maskable-512x512.png`.
- The Apple touch icon is an original 180×180 PNG generated by the repository's icon script.

### App Opens in Browser Instead of Standalone
- Remove the home screen shortcut and add it again from the production URL.
- Confirm the manifest uses `display: standalone`.
- Confirm the app is launched from the installed home screen icon, not from a browser bookmark.

### Data Missing After Browser Cleanup
- Browser cleanup may remove IndexedDB data.
- Restore from a previously exported backup JSON file.
- If no backup exists, the local collection may not be recoverable from the app.

## Manual Verification Checklist
- Build the production app with `npm run build`.
- Serve the production build over HTTPS or use the final production preview URL.
- Android Chrome:
  - App install prompt appears, or **Install app** / **Add to Home screen** is available.
  - Installed app opens in standalone mode.
  - Data persists after closing and reopening the installed app.
- iOS Safari:
  - **Add to Home Screen** works.
  - Icon appears correctly on the home screen.
  - App opens in standalone web app mode.
  - Data persists after closing and reopening the installed app.
- Offline:
  - App shell loads after the first successful visit.
  - Existing local data remains available.

## Deployment Path Note
This repo does not currently set Vite `base`. That is compatible with root-domain deployments. If the app is deployed under GitHub Pages at `/acg_app/`, the project may need `base: '/acg_app/'` so built asset URLs resolve under that subpath.

Do not add or change `base` without confirming the deployment target. A root deployment and a GitHub Pages subpath deployment need different URL assumptions.
